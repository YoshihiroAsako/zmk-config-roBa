# 共通ルール(全リポジトリ同一)

## 0. ファイル構成の標準

- AI エージェントへの指示の正本は各リポジトリの `AGENTS.md`(本ブロックはその共通層)。`CLAUDE.md` と `.cursor/rules/*.mdc` は「`AGENTS.md` を読んで従うこと」という誘導のみを書き、内容を複製しない。
- 本ファイルは共通層+固有層あわせて **150 行以内**を目標とする。超過は AI が自発的に解消せず、`.claude/skills/` または `docs/` への移動案を報告で提案する。
- 作業状態は `NextAction.md`(リポジトリ直下)、設計検討は `docs/design/`、完了済みの履歴は `Archive.md` に置く。
- 各リポジトリの固有層には「検証」節を置き、ビルド・テスト・lint・起動確認の実行コマンドを列挙する。§4 の検証は、まずこの節のコマンドで行う。
- ルールが競合する場合の優先順位: §6 安全 > §2 承認ゲート > §1 NextAction統制 > §5 スコープ遵守 > §4 実装・修正の質 > §3 トークン節約。スコープ外に及ぶ根本原因を発見した場合は、修正せず報告して指示を待つ。

## 1. NextAction.md の運用

- フォーマットは 4 部構成: **現在地 / 決定済み事項 / Next Actions / 保留・未決**。
- 作業開始時に `NextAction.md` を読む。存在しない場合は自発的に作成せず、その旨を報告して作業を進める(作成もユーザー指示時のみ)。長い参照先は必要なセクションだけ辿る。
- **【重要】AI が `NextAction.md` を自発的に更新することを禁止する。** 作業完了後も書き換えず、作業結果はチャットでの報告のみ行う。
- 更新は、ユーザーが NextAction.md の更新を明示的に指示したときのみ実施する(定型句は「**NextAction更新**」。同義の明確な指示も可)。更新時は、直前の作業の完了・決定・承認・却下のみを各セクションに反映し、**却下された作業は「保留・未決」へ移す**。指示で言及されていない項目は変更しない。
- 更新時、「決定済み事項」「保留・未決」は**追記のみ**とし過去の記録を書き換えない(訂正は新しい項目の追記で上書きする)。「現在地」「Next Actions」は現状を反映して書き直してよい。
- 「NextAction更新」の指示時にあわせて、クローズ済みの項目(完了した Next Action、決着した保留事項)を `Archive.md` へ**末尾追記**で移す。リポジトリ固有層で別の履歴ファイルが指定されている場合はそちらを使う。Archive の既存エントリは書き換え・並べ替え・削除しない。
- Archive への追記は「## YYYY-MM-DD: (件名)」の見出し+本文の形式で行う。

## 2. 設計ドキュメント分岐

次のいずれかに該当する作業は、**実装前に** `docs/design/` に検討ドキュメントを作成し、**ユーザーの明示的な承認(「承認」「その案で進めて」等、当該ドキュメントへの言及を伴うもの)を待ってから**実装する:

1. 一連の依頼で合計 3 ファイル以上の変更(新規作成を含む。NextAction.md・Archive.md・docs/ 配下の記録類は数えない)。作業を分割しても合計で判定する。
2. 新規外部依存(ライブラリ・サービス・ツール)の追加
3. データ構造(スキーマ・保存形式・API 契約)の変更
4. 複数の実装方針があり、どれを選ぶかで変更ファイル・外部依存・データ構造のいずれかが変わる場合(同一方針内の書き方の差は該当しない)。該当しないと判断した場合も、着手報告時に「採用する方針と、検討した代替案」を1行で述べること。

検討ドキュメントの構成は「背景 / 選択肢と比較 / 推奨案 / 影響範囲 / 未決事項」とする(雛形: design-doc テンプレート)。承認・却下を受けたら、ドキュメントの「状態」欄を更新してよい。

ゲートに該当しない作業も、編集前に「変更対象ファイルの一覧と各ファイルの変更内容(各1行)」を宣言してから着手する。宣言にないファイルへの変更が必要になったら、編集前に宣言へ追記して報告する。

## 3. トークン節約原則

- ファイルを全読みする前に、grep・ディレクトリ一覧等で構造を把握してから必要箇所だけ読む。
- 同一ファイルの不要な再読み込みをしない。
- 指示されていないファイルの探索をしない。ただし §2 の事前調査・§4 の根本原因特定に必要な探索はこの限りではなく、目的に必要な最小範囲(作業対象が import・参照するファイル、エラーメッセージやスタックトレースに現れるファイル)で行う。
- 会話に長い引用を貼らず、要点とファイルパスで共有する。ただしエラーメッセージ・テスト失敗の出力は要約せず、該当行の原文を報告に含める(無関係な前後の出力は省いてよい)。

## 4. 実装・修正時の原則

- 実在を確認していない API・関数・設定キー・CLI オプションを使わない。使用前に、リポジトリ内の既存の使用例・型定義・`node_modules` 内のドキュメントのいずれかで確認する。
- バグ修正は、症状への対処の前に**根本原因を特定する**。
- 修正方針を宣言してから、一貫して実施する。同一箇所への修正が 2 回続けて検証に失敗したら、追加の試行をせず、状況・試した内容・次の仮説を報告して指示を待つ。方針が誤りと判明したら、その時点で報告して方針を立て直す。
- 検証(テスト・ビルド・実機確認)は、実行可能な手段がある限り実施する。実施できなかった場合は、試した手段と失敗した理由を報告に明記する。トークン・時間の節約は省略の理由にならない。

## 5. 言語・様式

- 応答・ドキュメントは**日本語**。コード・識別子は英語。**コミットメッセージは日本語**。
- **結論を先に**述べ、根拠・詳細を後に添える。
- 指示されていないスコープ拡大をしない(機能追加・リファクタリング・最適化・体裁の一括変更を含む)。
- 不確実な内容は「推測」「要確認」と明記し、根拠のない数値・事実を提示しない。
- 着手前に依頼の解釈が複数あり、どれを選ぶかで成果物が実質的に変わる場合は、推測で進めず質問する。成果物に影響しない軽微な不明点は、置いた仮定を明示して進めてよい。

## 6. 安全

- ユーザーや他エージェントの未コミット変更を勝手に戻さない・上書きしない。
- 秘密情報(鍵・トークン・パスワード・個人情報)をコミットしない。ローカル秘密は `.env` 系(Git 管理外)に置く。
- 破壊的操作(ユーザー管理下のファイルの削除、`git reset --hard`、force push、履歴改変等)はユーザー承認を得てから行う。AI が同一作業内で作成した一時ファイルの削除は承認不要。
- `git commit` / `git push` はユーザーが指示したときのみ実施する。

## 7. 報告フォーマット

ファイル変更または複数ステップの調査を伴う作業の完了時、最終報告には次の 4 点を含める。単発の質問への回答には適用しない:

1. 今回やったこと
2. 検証結果(未検証の場合はその理由)
3. 次にやることの案(NextAction.md には書かず、報告として提示)
4. ユーザーの確認・判断が必要な点

# zmk-config-roBa — 固有ルール

## プロジェクト

- roBa 分割キーボードの ZMK Firmware 設定リポジトリ。MCU: `seeeduino_xiao_ble`(nRF52840)。ZMK は `config/west.yml` で `v0.3-branch` に固定。トラックボールドライバ: `kumamuk-git/zmk-pmw3610-driver`。

## ハードと設定の対応(誤解しやすい点)

- 物理配置と Kconfig フラグは別概念。**roBa_L = エンコーダー側 / roBa_R = トラックボール(PMW3610・SPI)+ ZMK Studio 側**。
- `CONFIG_EC11` と `CONFIG_ZMK_POINTING` は左右両方の `.conf` に**意図的に**存在する。片側にしかハードが無いことを理由に削除しない。
- 左右対称にすべき設定(バッテリー報告・split BLE 等)と、片側専用の設定(PMW3610・SPI・Studio・エンコーダー)を区別する。

## 変更禁止・維持事項

- `zephyr/` と `.west/` は編集しない。`config/west.yml` の revision は指示がない限り変更しない。
- `build.yaml` の roBa_R `snippet: studio-rpc-usb-uart`、`roBa_R.conf` の `CONFIG_ZMK_STUDIO=y` / `CONFIG_ZMK_STUDIO_LOCKING=n` を維持する。
- 既存の番号付きレイヤー(`layer_6` 等)を自動リネームしない。ピン番号は配線変更の意図がある時のみ変更。

## ワークフロー

- keymap 変更: `config/roBa.keymap` 編集 → 見た目が変わる場合は `keymap-drawer/roBa.yaml` 更新+`roBa.svg` 再生成を同一コミットに含める(コマンドと DSL の詳細は `.cursor/rules/zmk-keymap.mdc` と `docs/viewer-to-device-guide.md`)。
- 環境: Windows + PowerShell 前提。`>` リダイレクトは UTF-16 になるため、他ツールが読むファイルは UTF-8 を明示する。

## 検証

- ローカルビルドは前提にしない。push して GitHub Actions の成功を確認し、生成された `.uf2` で書き込む。DeviceTree ビルド成功は最低限の確認に過ぎない。
- ローカル viewer の動作確認は `tools/roba-keymap-viewer` の起動スクリプト(`start-roba-viewer.cmd`)で行う。

## 参照先(必要なときだけ読む)

- keymap DSL・roBa 固有注意: `.cursor/rules/zmk-keymap.mdc`(config/*.keymap 編集時に自動適用)
- ローカル viewer(tools/roba-keymap-viewer)の使い方: `docs/viewer-to-device-guide.md`
- Studio 代替アプリ計画(**凍結中**・明示指示まで着手しない): `docs/studio_alt_app.md`
